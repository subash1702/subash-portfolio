How I work

I start with the decision the data needs to support, then work backwards to define what “correct” means. I build for repeatability: clear schemas, idempotent runs, and checks that fail loudly when something changes upstream.

I prefer systems that stay understandable as they scale — small, testable components, consistent naming, and documentation that a new teammate can follow.
