Nexora-NX — AI Knowledge Graph Builder

Nexora-NX converts unstructured text and PDFs into a structured knowledge graph by extracting entity–relationship triplets, building a directed graph model, and enabling explainable “ask-the-graph” queries backed by path evidence.

The problem

Most knowledge in real teams lives in documents: PDFs, reports, meeting notes, and scattered text. It’s readable by humans, but hard to analyze as a system. The moment you want to answer questions like “what connects X to Y” or “what entities matter most,” you end up manually scanning pages.

I built Nexora-NX to turn unstructured documents into a structured relationship map that you can explore, query, and explain without hand-writing rules for every new domain.

In simple terms: it turns messy text into a navigable intelligence map.

What the system does

  • Ingests raw text or PDFs
  • Extracts entities and relationships using LLM-driven triplet output
  • Validates output into a structured schema (to avoid “LLM mess”)
  • Builds a directed knowledge graph in NetworkX
  • Visualizes the graph and supports interactive exploration

What you can do with it

  • See relationship networks (who/what connects to what)
  • Find key entities using PageRank-style centrality
  • Ask: “How is X connected to Y?” via shortest-path reasoning
  • Generate explainable answers using path evidence

How I engineered reliability

The hardest part wasn’t drawing a graph. It was getting structured extraction that holds up across inputs. I treated the LLM output like a data source: schema-controlled, validated, and retried when needed.

That approach keeps the graph consistent and prevents the UI from becoming a “random output generator.”

Why this matters

This project is a system, not a demo: AI extraction feeds a data model (graph), analytics runs on top of it, and the final experience is interactive and explainable. It’s the kind of hybrid setup that’s useful for document intelligence, research workflows, and internal knowledge discovery.

What I’d improve next

I’d add stronger entity resolution (deduping near-duplicates), a lightweight persistent store for graphs (so graphs survive restarts), and evaluation metrics for extraction quality across domains.
