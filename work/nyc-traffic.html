NYC Traffic Volume Analytics Pipeline

NYC traffic data is messy in predictable ways: missing sensor streaks, inconsistent labels, timestamp issues, and location fields that don’t join cleanly. I built a pipeline that standardizes the data into reliable analytical tables so trends can be trusted across neighborhoods and time.

Questions this supports: Which corridors are getting worse month over month? How do patterns change during major events? Where do spikes repeat by time-of-day and borough?

The messy reality

Raw traffic datasets are rarely analysis-ready. Sensors drop readings, timestamps don’t line up, and the same road segment can appear under slightly different labels. If you don’t fix those issues early, the insights drift as soon as you zoom in.

What I did

  • Standardized timestamps and removed duplicates / impossible values
  • Normalized location fields so segments match reliably
  • Built clean analytical tables (facts + dimensions) for repeatable queries

Why it matters

  • Teams can compare apples to apples across weeks and neighborhoods
  • Dashboards stay stable as new data arrives
  • Modeling becomes meaningful because inputs are consistent

What I’d add next

A lightweight data quality layer that watches for missing sensor streaks and unusual distribution shifts, plus versioned schemas so downstream users always know what changed.
