Dual-Layer E-Commerce Data Platform

I built a platform that treats streaming metrics and batch reporting as two different jobs, with two different failure modes — then designed each layer to be reliable on its own and consistent together.

Why I built this

A lot of “real-time dashboards” look fast until the data gets messy: events arrive late, ordering breaks, and the numbers quietly drift. At the same time, finance and reporting teams need stable batch datasets that reconcile cleanly.

The goal was to build something that stays correct as volume grows — not just something that works on demo day.

Layer 1: real-time event processing

I processed high-frequency transaction events with time-window logic so the dashboard could show “what’s happening now” without waiting for nightly batches.

  • Sliding and tumbling windows for sales velocity and inventory triggers
  • Handling out-of-order and late events without corrupting aggregates
  • Alerts for unusual stock movement and sudden revenue spikes

Layer 2: reliable batch analytics

For reporting, I created a batch ingestion and transformation layer designed for retries, audits, and consistency.

  • Daily ingestion into a central warehouse model
  • dbt transformations to clean and standardize raw tables
  • Idempotent runs to avoid duplication during retries
  • Data quality checks for schema + business rules

What I’d improve next

I’d add a schema registry + contract tests at the producer boundary, and expand DQ checks into anomaly detection (volume spikes, null surges, and dimension drift).
