Project

Dual Layer E-Commerce Data Platform

I built this because most teams need two kinds of truth at the same time: fast operational signals and stable, auditable reporting. This project shows how I design for both without sacrificing correctness.

Architecture in one paragraph

Events stream through Kafka. A Python processor computes windowed metrics and stays correct when events arrive late. In parallel, daily logs land in PostgreSQL and dbt builds reporting tables that remain consistent across retries. Quality checks validate schema and business rules before anything reaches BI.

What I cared about

Late events, duplicates, and retries aren’t rare edge cases—they’re normal. I designed the pipeline to handle them cleanly so the outputs stay trustworthy, not just “working.”

Tech

Python, SQL, Kafka, PostgreSQL, dbt, Docker
