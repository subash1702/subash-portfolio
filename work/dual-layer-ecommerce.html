Dual Layer E-Commerce Data Platform

A practical split between real-time event processing and a reliable batch layer, designed so metrics stay fresh without sacrificing correctness.

Python Kafka PostgreSQL dbt Docker

Problem

E-commerce events arrive fast, out of order, and with occasional late arrivals. Teams still expect dashboards to update quickly and numbers to match finance reporting later.

Approach

I built two layers that complement each other. The streaming layer focuses on low-latency signals (velocity, inventory triggers, alerts). The batch layer produces decision-ready tables with strict transformations and quality checks.

Architecture (high level)

[Event Producers] -> Kafka -> [Stream Processor]
                               |  windowed metrics
                               |  late event handling
                               v
                            Postgres (serving tables) -> dashboards/alerts

[Daily Logs / Backfills] -> Ingestion -> Warehouse tables -> dbt models -> BI-ready marts
                                   |         (tests + DQ gates)
                                   v
                              reliable reporting

Key engineering decisions

Windowing was built around sliding and tumbling windows so the system can answer “what’s happening now” without constantly reprocessing everything. I also designed the pipelines to be idempotent, so retries don’t duplicate data.

On the batch side, dbt tests act as quality gates (schema, uniqueness, and business logic checks) before anything is treated as a trusted dataset.

Outcome

The streaming layer produces low-latency signals while the batch layer preserves correctness and auditability. It’s the same story teams face in real companies: speed and trust need different mechanics.
