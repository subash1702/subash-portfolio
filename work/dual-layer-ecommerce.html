Project

Dual Layer E-Commerce Data Platform

A two-layer system that supports real-time operational visibility and stable batch reporting, without data drift or duplication.

Why I built it

In e-commerce, the questions come in two speeds. Operations needs fast answers right now (sales velocity, inventory triggers). Finance and reporting need answers that are consistent and auditable. I built this platform so both can exist in the same system without one breaking the other.

Architecture in one paragraph

Transaction events stream through Kafka. A Python processor computes windowed metrics and handles late events so aggregates stay correct. In parallel, daily logs load into PostgreSQL and dbt models transform them into clean reporting tables. Quality checks validate schema and business rules so bad data is caught before it reaches dashboards.

Key decisions

Late and out-of-order events: I treated event time as the source of truth and designed window logic to update correctly when delayed events arrive.

Idempotent batch loads: retries are normal in production, so the batch layer is safe to rerun without duplicating results.

Quality checks as a first-class feature: schema and business rule validation prevents “silent wrong” reporting.

Tech stack

Python, SQL, PostgreSQL, Kafka, dbt, Docker

What I’d improve next

Add stronger observability (latency and quality dashboards), a schema registry, and automated backfills for late partitions.
