Dual Layer E-Commerce Data Platform

I built this to handle the two realities of commerce data: live events that need fast answers, and daily reporting that needs consistency. The goal was simple: keep the realtime layer responsive, and make the batch layer boring and correct.
Streaming + Batch, built to survive messy data

What I built

  • Realtime processing with windowed metrics (velocity, triggers)
  • Late and out-of-order event handling to keep outputs stable
  • Batch ingestion + dbt transforms for clean analytical tables
  • Idempotent jobs + DQ checks to avoid duplicates and bad loads

How I think

I design for failure first. If an event arrives late, the pipeline shouldn’t panic. If a job retries, results shouldn’t change. And if a metric looks weird, I want enough logging and checks to find the root cause quickly.
