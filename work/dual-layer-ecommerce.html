Focus Streaming + Batch reliability

Turning noisy events into decision-ready data.

I built this as a two-layer platform: a real-time processing layer for fast, time-windowed metrics, and a batch analytical layer that produces clean, auditable tables for reporting. The goal was simple: keep the dashboards fast, but make the numbers trustworthy.

What I built

  • A streaming pipeline with tumbling + sliding windows for near real-time business signals (sales velocity, inventory triggers).
  • Logic to handle out-of-order and late events so metrics don’t “jump” when data arrives late.
  • A batch ingestion + dbt transformation layer designed for repeatable, auditable reporting.
  • Idempotent pipeline behavior so retries don’t create duplicates.
  • Data quality checks that validate schema + business rules before anything reaches reporting tables.

How I think about it

I treat pipelines like products: reliability first, then speed. I try to make the “messy middle” predictable—clear contracts, defensive checks, and simple failure modes.
